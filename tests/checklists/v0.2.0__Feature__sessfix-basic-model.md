# Human Test Checklist: sessfix basic model

**Target version:** 0.2.0
**Ships with commit:** `feat: sessfix install/recover/uninstall + lock-after-rescue` (`763aa4d`)
**Companion automated tests:** 7 tests in `tests/test_version.py` (version scaffolding only). The sessfix behavior below is **PowerShell + real Windows session APIs (`tscon`, WTS, LSM events, NSSM service)** and is **not** covered by pytest — this checklist is the coverage for it.

## What this is

wtf-rdp rescues an RDP session that Windows has wedged/blocked by Local Session Manager (LSM) — reconnecting it with `tscon` **and locking it**, instead of the destroy-the-session "sign out" Windows otherwise forces. The `sessfix` kit ships the tools around a LocalSystem watchdog: `install`, `status`, `recover`, `uninstall`. This document is what a human walks through before shipping a release: it verifies the parts mocks can't — that a rescued session is genuinely **locked** (not an open desktop), that running work **survives**, that the watchdog **doesn't false-fire**, and that the service really runs as **SYSTEM**.

## Prerequisites

1. **Windows 10 or 11** (client or Server). The tools are Windows-only.
2. **Administrator** shell (installing/removing a service and running the rescue both need elevation).
3. wtf-rdp installed: `pip install wtf-rdp` (or `pip install -e .` from a clone). `rdp --version` should print 0.2.0.
4. **A throwaway local admin account** to use as the rescue *subject* (referred to below as `rdptest`) — you do **not** want to test rescues against your own working session until you trust it.
5. **Isolation / safety:**
   - The service installs as `wtf-rdp-sessfix` to `C:\ProgramData\wtf-rdp`. Use **`-TargetUser rdptest`** on install so the watchdog only ever acts on the throwaway account.
   - A second machine (or RDP client) is handy to drive the `rdptest` RDP session.
   - **Reset between sections:** `rdp uninstall -Purge` removes the service + install dir. Re-run `rdp install ...` to start clean.
6. On a first run you may want `-NssmPath <path-to-nssm.exe>` to skip the download while testing the flow; then separately test the real download path (Section 1.2).

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. If any fail, the feature is broken.

### HV.1 — `rdp install` produces a running LocalSystem service

Validates: the whole deploy → register → start path, as SYSTEM.

- [ ] **Steps** (elevated PowerShell):
  ```powershell
  rdp install -TargetUser rdptest -NssmPath C:\path\to\nssm.exe
  rdp status
  ```
- [ ] **Expected:** install prints "Registered 'wtf-rdp-sessfix' as LocalSystem", "Service status: Running"; `rdp status` shows the service **Running**.
- [ ] **Check:**
  ```powershell
  Get-CimInstance Win32_Service -Filter "Name='wtf-rdp-sessfix'" | Select State,StartName,StartMode
  Get-Content C:\ProgramData\wtf-rdp\watchdog.log -Tail 2
  ```
  `StartName = LocalSystem`, `StartMode = Auto`; the log's start line says **`identity=NT AUTHORITY\SYSTEM`**.
- [ ] **Why this matters:** if it isn't SYSTEM, the rescue can't `tscon` a cross-user session or lock it — the whole tool is dead.

### HV.2 — A rescued session comes back **LOCKED** (the security fix)

Validates: recovery never leaves an open desktop.

- [ ] **Steps:** with the service installed (`-SimulateWedgeFile C:\ProgramData\wtf-rdp\SIMULATE_WEDGE` set for a deterministic trigger — see 1.3), RDP in as `rdptest`, disconnect that session, then drop the sentinel file. Wait for the confirm window.
- [ ] **Expected:** `watchdog.log` shows `RESCUE OK: ... locked=True`; the `rdptest` session is `Active` on the console.
- [ ] **Check:** find rdptest's session id (`qwinsta`), then:
  ```powershell
  @(Get-Process LogonUI | Where-Object SessionId -eq <SID>).Count   # >0 == LOCKED
  ```
- [ ] **Why this matters:** the earlier design left the box **unlocked** after rescue — a recovered machine open on the console is worse than the disease.

### HV.3 — Running work SURVIVES the rescue

Validates: the rescue is non-destructive (reconnect, not sign-out).

- [ ] **Steps:** in the `rdptest` RDP session open Notepad and type text (don't save). Disconnect ungracefully. Trigger a rescue (sentinel, or `rdp recover`). Reconnect as `rdptest`.
- [ ] **Expected:** Notepad is **still open with your text**; the session's logon time is unchanged (`quser`).
- [ ] **Why this matters:** the entire point is to *not* lose work — a rescue that killed the session would be a regression to the Windows "sign out" behavior.

### HV.4 — The watchdog does NOT false-fire on a benign idle login screen

Validates: detection discrimination (console-connecting alone is not a wedge).

- [ ] **Steps:** with the service installed but **no** sentinel and **no** real wedge, leave the machine at its login screen with a normal disconnected `rdptest` session present. Wait > confirm window.
- [ ] **Expected:** `watchdog.log` shows **no** `RESCUE` line; the disconnected session is left alone.
- [ ] **Why this matters:** a watchdog that yanks any disconnected session onto the console whenever the machine is idle is a menace, not a guardian.

### HV.5 — `rdp recover` (manual) reconnects + locks on demand

Validates: the manual path + SYSTEM self-elevation.

- [ ] **Steps:** leave a stranded (disconnected) `rdptest` session, then (elevated): `rdp recover -TargetUser rdptest`
- [ ] **Expected:** prints `Recovered session <id>: reconnected, locked=True`; session is Active on console and **locked** (HV.2 check).
- [ ] **Why this matters:** `recover` is the human's escape hatch when the watchdog isn't installed or didn't catch it.

### HV.6 — `rdp uninstall` cleanly removes the service

- [ ] **Steps:** `rdp uninstall` then `rdp status`
- [ ] **Expected:** service removed; `rdp status` shows **NOT INSTALLED**. `Get-Service wtf-rdp-sessfix` errors (gone).

## Section 1: Install & service

### 1.1 Install with a provided NSSM

- [ ] **Steps** (elevated): `rdp install -TargetUser rdptest -NssmPath C:\path\to\nssm.exe`
- [ ] **Expected:** "Using provided NSSM", "Deployed watchdog + module to C:\ProgramData\wtf-rdp", service Running.
- [ ] **Check:** `C:\ProgramData\wtf-rdp` contains `Watch-RdpSession.ps1`, `WtfRdp.Sessions.psm1`, `nssm.exe`.

### 1.2 Install with the real NSSM download (checksum path)

- [ ] **Steps:** on a machine with no cached nssm: `rdp install -TargetUser rdptest`
- [ ] **Expected:** "Downloading NSSM (nssm.cc/release/nssm-2.24.zip)"; prints `NSSM SHA256 = <hash>`; extracts the correct arch (win64/win32); service Running.
- [ ] **Then pin it:** re-run with `-NssmSha256 <that hash>` and confirm "NSSM checksum verified." A **wrong** `-NssmSha256` must **abort** with "checksum mismatch".
- [ ] **Why this matters:** we deliberately don't ship the binary in the wheel; the download+verify is the trust boundary.

### 1.3 Test-hook install (deterministic wedge) for later sections

- [ ] **Steps:** reinstall with the sentinel hook and faster timings:
  ```powershell
  rdp install -TargetUser rdptest -NssmPath C:\path\to\nssm.exe -PollIntervalSec 8 -WedgeConfirmSec 15
  # then add the sentinel param (NSSM):
  & C:\ProgramData\wtf-rdp\nssm.exe set wtf-rdp-sessfix AppParameters '-NoProfile -ExecutionPolicy Bypass -File "C:\ProgramData\wtf-rdp\Watch-RdpSession.ps1" -PollIntervalSec 8 -WedgeConfirmSec 15 -TargetUser rdptest -SimulateWedgeFile "C:\ProgramData\wtf-rdp\SIMULATE_WEDGE"'
  Restart-Service wtf-rdp-sessfix
  ```
- [ ] **Note:** production must NOT set `-SimulateWedgeFile`. Confirm the default install has no sentinel param.

### 1.4 Idempotent re-install

- [ ] **Steps:** run `rdp install ...` twice.
- [ ] **Expected:** the second run stops+removes the existing service first (no "already exists" error) and ends Running.

### 1.5 Non-admin is refused

- [ ] **Steps:** run `rdp install` from a **non-elevated** shell.
- [ ] **Expected:** clean error "Administrator required..." — no partial install.

## Section 2: Detection discrimination

### 2.1 No false-fire on idle login screen (see HV.4)

- [ ] Covered in HV.4. Also confirm `watchdog.log` logs the benign case quietly (a "no recent Event 36 -> benign" line at most), not a rescue.

### 2.2 No false-fire on a normal disconnected session

- [ ] **Steps:** RDP in as `rdptest`, disconnect **gracefully** (close the RDP client normally). Leave it. No sentinel.
- [ ] **Expected:** the session stays `Disc`; no rescue. (Normal disconnects are not wedges.)

### 2.3 Ambiguity guard

- [ ] **Steps:** with `-TargetUser` **empty** and two different users' disconnected sessions present, trigger a wedge.
- [ ] **Expected:** `watchdog.log` logs "MULTIPLE stranded sessions ... ambiguous -> set -TargetUser; no-op" — it does NOT guess.

## Section 3: Rescue + lock (security-critical)

### 3.1 Deterministic rescue + lock via sentinel (see HV.2)

- [ ] **Steps:** stranded `rdptest` + drop `C:\ProgramData\wtf-rdp\SIMULATE_WEDGE`. Watch `watchdog.log`.
- [ ] **Expected sequence:** `WEDGE CANDIDATE` → (confirm window) → `RESCUE: ... reconnecting + locking` → `RESCUE OK: ... locked=True`.
- [ ] **Check:** LogonUI present in the session (locked); explorer also present is fine (the desktop is loaded *behind* the lock screen).
- [ ] **Cleanup:** delete the sentinel afterward: `[System.IO.File]::Delete("C:\ProgramData\wtf-rdp\SIMULATE_WEDGE")`.

### 3.2 Lock genuinely blocks access

- [ ] **Steps:** after a rescue, look at the physical console (or `mstsc /admin`).
- [ ] **Expected:** a **lock screen / credential prompt**, not the user's live desktop.

### 3.3 Cooldown / no thrash

- [ ] **Steps:** with the sentinel left in place, watch across several poll cycles after a successful rescue.
- [ ] **Expected:** it does not re-rescue the same session every cycle (per-session cooldown); once the console is Active it stops treating it as a wedge.

## Section 4: `rdp recover` (manual)

### 4.1 Auto-pick the single stranded session (see HV.5)

### 4.2 Explicit `-SessionId`

- [ ] **Steps:** `rdp recover -SessionId <id>` for a specific stranded session.
- [ ] **Expected:** that session reconnected + locked.

### 4.3 Nothing to do / ambiguous

- [ ] **Steps:** run `rdp recover` with no stranded session; then with two (no `-TargetUser`).
- [ ] **Expected:** "No stranded ... session to recover." / "Multiple stranded sessions ... pass -SessionId or -TargetUser." — no action, no error crash.

## Section 5: `rdp uninstall`

### 5.1 Remove service (see HV.6)

### 5.2 Purge

- [ ] **Steps:** `rdp uninstall -Purge`
- [ ] **Expected:** service gone AND `C:\ProgramData\wtf-rdp` deleted (or a clean message if a file is in use).

## Section 6: Real-wedge induction (best-effort — the honest hard part)

Reproducing a *real* LSM block / Event 36 is an intermittent race; the deterministic sentinel path (Section 3) is the reliable test. Attempt a real one:

- [ ] **Steps:** RDP in as `rdptest` and cause an **ungraceful** disconnect during connect/reconnect — e.g. kill the network (block TCP+UDP 3389 to the target) *mid-connect*, or kill the `mstsc` process during the login transition. Repeat a few times.
- [ ] **Check for the signal:** in the LSM Operational log look for **Event 36** (`CsrConnected`→`EvCsrInitialized`, error `0x800708CA`):
  ```powershell
  Get-WinEvent -FilterHashtable @{LogName='Microsoft-Windows-TerminalServices-LocalSessionManager/Operational'; Id=36; StartTime=(Get-Date).AddMinutes(-5)}
  ```
- [ ] **Expected (if a real wedge forms):** the running watchdog (in production mode, **no** sentinel) detects console-connecting + Event 36, confirms, and rescues + locks — same `watchdog.log` sequence as 3.1.
- [ ] **If it won't reproduce:** that's expected/known — note it and rely on the sentinel path. (Open question in the design: whether some blocks lack Event 36 and need a secondary signal.)

## What the automated tests DON'T cover (this checklist is the coverage)

- The lock actually engaging (`LogonUI` in the session) — a real Win32 `CreateProcessAsUser` into another session; no mock proves the box is locked.
- `tscon` reconnecting a cross-user session as SYSTEM (needs `SeTcbPrivilege`).
- Running work surviving an ungraceful disconnect + rescue.
- Detection discrimination against benign idle/disconnected states (real LSM event stream).
- NSSM download + checksum verification + service registration as LocalSystem.
- The single-interactive-session behavior of client Windows.
- Error-message clarity for the non-admin / ambiguous / nothing-to-do paths.
- Real LSM Event 36 induction (intermittent).

## Reporting issues found

1. Note the section + step (e.g. "HV.2").
2. Capture the exact command, `watchdog.log` tail, and `qwinsta` output.
3. Save `C:\ProgramData\wtf-rdp\watchdog.log`.
4. File an issue referencing this checklist and commit `763aa4d`.

## Cleanup

- [ ] `rdp uninstall -Purge` (removes service + `C:\ProgramData\wtf-rdp`).
- [ ] Delete any leftover `SIMULATE_WEDGE` sentinel.
- [ ] Log off / remove the throwaway `rdptest` account if it was created only for this run.
- [ ] Confirm your own session/work is intact.
