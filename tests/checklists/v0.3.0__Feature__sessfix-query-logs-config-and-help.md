# Human Test Checklist: sessfix query / logs / config + universal `-h`

**Target version:** 0.3.0
**Ships with commit:** `feat: sessfix query/logs/config, universal -h, verified NSSM pin` (TBD)
**Companion automated tests:** 23 tests total — `tests/test_cli.py` (16: kit registration, per-tool manifest/runtime integrity, `-h` handling, `rdp -h` epilog) + `tests/test_version.py` (7). The live-service behavior below (`nssm get/set`, `Restart-Service`, WTS enumeration, log tailing) is **PowerShell against a real Windows service** and is **not** covered by pytest — this checklist is the coverage for it.

## What this is

0.3.0 rounds out the `sessfix` kit with three tools — `query` (read-only session diagnostic), `logs` (watchdog log viewer), `config` (show/change the watchdog's runtime parameters) — and makes **every** tool answer `-h` / `-help` / `--help`. This document verifies the parts mocks can't: that `config` reads NSSM's UTF-16 output correctly, that a config change actually reaches the running watchdog, that `query` classifies sessions the same way the watchdog does, and that no `rdp <tool> -h` ever dies with a PowerShell "parameter cannot be found" error.

## Prerequisites

1. **Windows 10 or 11**, **Administrator** shell for anything that installs/changes the service.
2. wtf-rdp installed (`pip install -e .` from a clone). `rdp --version` prints `0.3.0`.
3. A checksum-verified NSSM will download automatically; to test offline pass `-NssmPath <nssm.exe>`.
4. **Isolation / safety:** always install with **`-TargetUser <throwaway-or-nonexistent-name>`** so the watchdog can never act on your real session. The validation run for this release used `-TargetUser wtfrdpnobody` (a name that matches no account) so rescues could never fire.
5. **Reset between sections:** `rdp uninstall -Purge` removes the service + `C:\ProgramData\wtf-rdp`.

## High-Value Verification (~5 minutes)

If you only have 5 minutes, run these. If any fail, the release is broken.

### HV.1 — Every tool answers `-h`, `-help`, and `--help`

Validates: the `Alias('h')` switch + `ValueFromRemainingArguments` catch-all on all seven tools, and the top-level epilog.

- [ ] **Steps** (any shell):
  ```
  rdp -h
  rdp status -h
  rdp query -h
  rdp install -h
  rdp recover -h
  rdp logs -h
  rdp config -h
  rdp uninstall -h
  rdp config --help
  rdp status -help
  ```
- [ ] **Expected:** `rdp -h` lists all seven tools + a "getting started" block; each `rdp <tool> -h/-help/--help` prints that tool's usage text.
- [ ] **Expected NOT:** none of them print `A parameter cannot be found that matches parameter name 'h'` (the 0.2.x bug this fixes).
- [ ] **Why this matters:** `-h` is the first thing a new user types; if it errors, the tool looks broken on contact.

### HV.2 — `config` reads the live watchdog parameters correctly (not "(default)")

Validates: the UTF-16/NUL-strip fix on `nssm get AppParameters`.

- [ ] **Steps** (elevated):
  ```powershell
  rdp install -TargetUser wtfrdpnobody -PollIntervalSec 20 -WedgeConfirmSec 40 -NssmPath C:\path\to\nssm.exe
  rdp config
  ```
- [ ] **Expected:** `rdp config` shows `PollIntervalSec : 20`, `WedgeConfirmSec : 40`, `TargetUser : wtfrdpnobody` — **not** "(default)" / "(any)".
- [ ] **Why this matters:** before the fix, NSSM's UTF-16 output was captured with interleaved NUL bytes and every value parsed as "(default)", making `config` useless.

### HV.3 — `config` set reaches the running watchdog

Validates: `Set-Param` regex, `Restart-Service` restart, and that the watchdog restarts with the new value.

- [ ] **Steps** (elevated, service installed):
  ```powershell
  rdp config -PollIntervalSec 30
  rdp config
  rdp logs
  ```
- [ ] **Expected:** the set prints `Now: PollIntervalSec=30 WedgeConfirmSec=40 TargetUser=wtfrdpnobody` and `Service status: Running`; `rdp config` confirms `30` while **preserving** `40` and `wtfrdpnobody`; `rdp logs` shows a fresh watchdog start line with `poll=30s`.
- [ ] **Expected NOT:** no `Unexpected status SERVICE_PAUSED` error (the 0.2.x-era `nssm restart` symptom).
- [ ] **Why this matters:** a config tool that silently fails to apply, or corrupts `AppParameters` and crashes the service, is worse than none.

### HV.4 — `query` enumerates sessions and classifies them

Validates: the shared `Get-WtfRdpSession` / `Get-WtfRdpWedgeEventSid` path.

- [ ] **Steps:** `rdp query` then `rdp query -Json`
- [ ] **Expected:** a table with `Id / WinStation / User / State / Stranded / WedgeSignal`; your live console session shows `State=Active`, `Stranded=False`, `WedgeSignal=False`. Session 0 (`Services`, no user) is **not** flagged Stranded. `-Json` emits valid JSON with `sessions` + `wedgeEventSids`.
- [ ] **Why this matters:** `query` is the human-facing view of the exact signals the watchdog acts on; if its classification drifts from the watchdog's, the diagnostic lies.

## Section 1: `query` details

### 1.1 Stranded vs. wedge distinction
- [ ] **Steps:** with a normal disconnected session present (RDP in as the throwaway user, then disconnect), run `rdp query`.
- [ ] **Expected:** that session shows `Stranded=True` but `WedgeSignal=False` (a normal disconnect is not a wedge). Footer note: "N stranded ... none show a wedge signal (this is normal)."
- [ ] **Check:** `-WedgeLookbackMin 120` widens the LSM Event 36 scan window without error.

### 1.2 No admin required
- [ ] **Steps:** run `rdp query` from a **non-elevated** shell.
- [ ] **Expected:** the session table still prints (the wedge-event scan may be limited without admin, but the command must not throw).

## Section 2: `logs` details

### 2.1 Tail / Full / Follow / Service
- [ ] **Steps** (service installed and has run):
  ```powershell
  rdp logs                 # last 40 lines
  rdp logs -Tail 5
  rdp logs -Full
  rdp logs -Service        # NSSM stdout/stderr logs
  ```
- [ ] **Expected:** tail shows recent lines; `-Full` prints the whole file; `-Service` prints the `service-stdout.log` / `service-stderr.log` sections (or "(not present)").
- [ ] **Follow:** `rdp logs -Follow` streams and blocks until Ctrl-C (test briefly).

### 2.2 Graceful when nothing is installed
- [ ] **Steps:** with no service installed (`rdp uninstall -Purge` first), run `rdp logs`.
- [ ] **Expected:** a friendly "No watchdog log at ... (is the service installed ...)" message, **not** a stack trace.

## Section 3: `config` details

### 3.1 Read-only without setters
- [ ] **Steps:** `rdp config` (no flags), `rdp config -Json`
- [ ] **Expected:** current values printed / valid JSON. No admin required to *read*.

### 3.2 `-TargetUser` / `-ClearTargetUser` round-trip
- [ ] **Steps** (elevated):
  ```powershell
  rdp config -TargetUser someuser
  rdp config                       # TargetUser: someuser
  rdp config -ClearTargetUser
  rdp config                       # TargetUser: (any / unrestricted)
  ```
- [ ] **Expected:** target user is set, then cleared; the `-File "...Watch-RdpSession.ps1"` prefix in `AppParameters` is preserved throughout (the watchdog still starts). **Caution:** `-ClearTargetUser` makes the watchdog act on *any* stranded session — only do this on a machine where that is safe.

### 3.3 Setting requires admin
- [ ] **Steps:** from a **non-elevated** shell, `rdp config -PollIntervalSec 25`
- [ ] **Expected:** throws "Administrator required to change service config."

### 3.4 Missing service
- [ ] **Steps:** with no service installed, `rdp config`
- [ ] **Expected:** "nssm.exe not found in ... run 'rdp install' first." (graceful, no stack trace).

## Section 4: install robustness (regression)

### 4.1 Verified NSSM download by default
- [ ] **Steps:** on a machine with no cached NSSM, `rdp install -TargetUser wtfrdpnobody`
- [ ] **Expected:** downloads `nssm-2.24.zip`, prints "NSSM checksum verified." (the SHA256 is pinned by default now), registers + starts the service.
- [ ] **Note:** `nssm.cc` occasionally returns HTTP 503; if so, re-run or pass `-NssmPath`. `-Purge` deletes the cached NSSM, so the next install re-downloads.

## What the automated tests DON'T cover

- Real NSSM `get`/`set` and its UTF-16 output (the exact bug HV.2 guards).
- `Restart-Service` actually restarting the watchdog with new params (HV.3).
- Live WTS session enumeration and LSM Event 36 correlation (`query`).
- Log tailing / following against a real file being written by the service.
- PowerShell parameter binding of `-h`/`-help`/`--help` in a real shell (pytest only checks the source pattern is present).
- Terminal rendering of the `query` table and help text on the Windows console codepage.
